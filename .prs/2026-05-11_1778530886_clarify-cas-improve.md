# PR

## User Summary (do not modify)

In the previous PR, we made clarify_public_api produce cas files, logging the question and answer.

In this PR, we build `codalotl docs improve-from-clarify`. This command:
- finds "in-play" cas files from clarify_public_api.
    - files where the hash matches the package's current hash.
    - AND files that are uncommited (unstaged and staged).
    - AND, files which were added in this git branch (so even if the hash doesn't match, it's presumably part of the current workstream, and not wildly out of date)
- for each one:
    - potentially improve documentation on that package (LLM decides if q/a is relevant/is worthwhile to improve. if so, edit doc comments. otherwise, don't) (each one can have multiple q/a's. do this for each one.)
    - delete the cas file - we no longer need it

### Phase 1: Clarify CAS record discovery [DONE]

#### Package `internal/gocas/casclarify`

- [DONE] Add public API to list in-play clarify records in the current git workstream.
- [DONE] An in-play record belongs to the current workstream when its CAS file is uncommitted, staged, or added on the current branch.
- [DONE] Current-hash records qualify when they belong to the current workstream; branch-added records also qualify after hash drift.
- [DONE] Add public API to delete a selected clarify CAS record after successful processing.

### Phase 2: Documentation improvement from clarifications

#### Package `internal/docubot`

- Add a docubot operation that accepts clarification Q/A entries for a package.
- Reuse the package's existing doc editing behavior and let the LLM decide whether each Q/A is useful enough to incorporate.
- No-op when the Q/A is irrelevant, already reflected in existing docs, or not worth documenting.

### Phase 3: CLI command

#### Package `internal/cli`

- Add `codalotl docs improve-from-clarify`.
- Command finds in-play clarify CAS records for the current module, groups/applies each record to its target package, and deletes each processed record after a successful package pass.
- Use effective model/configured reflow width, and print a concise summary of records processed, records deleted, and doc changes.
