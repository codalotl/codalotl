To see how this is done, you can inspect /Users/jonathannovak/leaf/code/others/opencode or /Users/jonathannovak/leaf/code/others/codex

The below is handwavy and is intended to show architecture, not specific method names or parameters.

New package: `internal/subscriptions/openaisub`
- contains the code that knows how to deal w/ openai subscriptions
- calls methods on llmmodel to set set subscription info in some sort of provider agnostic way

LLMStream does NOT depend on `internal/subscriptions/openaisub`. Instead it calls llmmodel to get provider subscription info if available.
- note: for openai, subscription info needs zdr (nostore). It also requires setting systeminstructions or whatever on root request, vs using an input array item.
    - so if llmstream sees we're using the subscription (which we do, over api keys, if the subscription is "logged in"), it forces nostore=true.

CLI:
- `codalotl auth openai login`
- `codalotl auth openai login --no-browser`
- `codalotl auth openai logout`
- `codalotl auth openai status`

CLI startup (for TUI/exec, for instance) calls ~ `openaisub.Init()` (or something better named, which):
- sees if `~/.codalotl/openai_auth.json` exists (path may be injected? see how we deal w/ this global config path elsehwere)
- If so and is valid, calls `llmmodel.SetSubscription(the info)`

Validation:
- `~/.codalotl/openai_auth.json` already exists (and is valid, i think) from a prior attempt at this
- use `go run . exec` to make sure this actually works.
- run the above with an unset openai api key, so that we know for sure that openai subscription auth is working
- Make sure llmstream.log logs the path for http requests, so we can see from there that this is actually being triggered.
