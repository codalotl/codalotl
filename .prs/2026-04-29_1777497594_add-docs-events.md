# PR

## User Summary (do not modify)

Context/Problem:
- In internal/docubot, AddDocs exists, which asks LLM to document code. Can take along time to run or slow models and big packages/lots of docs to update. Multiple LLM calls. Does NOT use internal/agent based infra.
- It prints things out to stdout, and returns ([]*gopackagediff.Change, error)
- This is ~fine in isolation, I want to use this in multiple contexts:
    - in normal CLI mode (internal/cli), it's ok
        - even still: CLI has configurable io.Writer for stdout, so we'd rather write to that.
    - but I want to plug this into TUI eventually. In TUI, i am not sure what the exact output shape should be, but i know it's not writing to stdout.
        - Important: writing to stdout in TUI will hard-break the TUI.
- running AddDocs actually takes a long time (minutes). In context of TUI, i'd like to somehow stream updates to user. Dont want "frozen TUI" for 5 minutes and then a big dump of updated docs.

Options:
- "just write to a configurable writer".
    - this might be good enough and work in practice really well
    - at the same time, though, it's not a very configurable UI.
        - For instance, what if TUI doesn't want to display all the noise it generates now? what if it only wants print results as they come in?
- "use events on a channel"
    - channel could either be returned (AddDocs internally calls `go runAddDocs(ch); return ch`)
    - or channel could be injected: AddDocs is still synchronous. If a channel is supplied in options, we send updates there.

Overall, from the perspective of `internal/docubot`, and where it sits in the layering, I think supporting both is actually reasonable:
    - Options include a writer. if nil, we just use stdout.
    - Options include a channel where we can emit semantic events.

In this PR:
- AddDocs supports a configurable writer. Put it in base options. Just make the whole package use that (not just AddDocs).
- Make cli supply that.
- we can potentially add the semantic event channel in a future PR. Not this one.

## Plan

### Package internal/docubot
- Update `BaseOptions` so docubot operations can emit user-facing progress to a configurable writer; nil keeps current stdout behavior.
- Route existing `userMessagef` / progress output through that writer across the package, while keeping structured logging on `health.Ctx` / logger.
- Add focused tests around injected-writer behavior and default fallback behavior.

### Package internal/cli
- Pass CLI output writer into `docubot.BaseOptions` for `codalotl docs add`.
- Keep docs-add summary behavior, but ensure all docubot progress respects `RunOptions.Out`.
- Add/adjust focused CLI tests as needed.
