# PR

## User Summary (do not modify)

```go
// DiscoverModules returns the Go modules discovered from root.
//
// If root is a file, discovery starts from its parent directory. The returned
// modules are sorted by absolute module path.
//
// Discovery follows Go's ordinary project-discovery semantics. If a Go
// workspace applies to root, the workspace's explicitly listed modules are
// returned. Otherwise, discovery recursively searches below root for go.mod
// files, using the same broad-recursive exclusions as Go package patterns:
// vendor directories, testdata directories, directories whose names begin with
// ".", and directories whose names begin with "_" are not descended into.
//
// The root directory itself is always considered before recursive exclusions
// are applied, so calling DiscoverModules directly on a module directory returns
// that module even if the directory name would otherwise be skipped during a
// parent-root recursive search.
func DiscoverModules(root string) ([]*Module, error)
```

Implement and test that in gocode.

Use in internal/cli. Some things in internal/cli have different implemented semantics. We want `codalotl cas ls-stale` to use this. Read product-spec about how the root/sandbox dir relates to module discovery. The user would expect this to translate to, for instance, `codalotl cas ls-summary`. They'd in general expect this to be how module discovery works. So let's normalize to this, provided it makes sense. Flag for me anything else you modify or decide to not modify.

Validate by running ~ `go run . cas` commands and any other things you modify to make sure it works as you expect.

## Decisions

- CLI audit after implementation:
  - [DONE] Strong follow-up candidate: `codalotl spec status`. It is a no-arg package-enumerating status command, so repo/workspace module discovery is more consistent with `cas ls-summary`/`cas ls-stale`.
  - Possible but not clearly in-scope: `codalotl context packages`. Its current docs and `gocodecontext.PackageList` API are explicitly current-module-oriented, with direct dependency listing; using repo/workspace discovery would be a separate design change.
  - Not good candidates: single-package commands (`context public`, `context initial`, `docs add`, `docs fix`, `spec diff`, `cas get`, `exec --package`) because they intentionally resolve one package; `spec ls-mismatch` because it accepts an explicit Go package pattern; `docs reflow` because it works on explicit file/dir paths.
