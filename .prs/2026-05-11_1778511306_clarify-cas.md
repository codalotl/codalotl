# PR

## User Summary (do not modify)

We're going to change how clarify improves documentation. I dislike how files "randomly" appear to sprout new comments. Surprising to the user.

New solution:
- clarify does not autoimprove docs
- instead, it writes a CAS file attached to the target package. Inlcuded metadata:
    - question
    - answer
    - package that originated the question
    - anything else?

A future PR (not this one) will build a tool/solution to analyze these cas files to actually improve docs.

## Plan

### Phase 0: Clarify CAS schema

#### Package `internal/gocas/casclarify`
- Create a thin CAS wrapper package for clarify records attached to a `*gocode.Package`.
- Store a versioned record under one namespace, with append semantics so multiple clarification questions for the same package content are preserved in one CAS payload.
- Each entry should include the question, answer, origin package, target package, and identifier. Best-effort git metadata remains owned by `internal/gocas`.

### Phase 1: `clarify_public_api` records CAS instead of editing docs

#### Package `internal/tools/pkgtools`
- Remove the automatic docs-improvement subagent from `clarify_public_api`; the tool must not edit target source files after answering.
- After a successful clarification for a writable sandbox package, append a clarify CAS entry to the target package.
- Resolve the target package and the caller/origin package to stable package identities when building the CAS entry.
- Authorize CAS writes under the sandbox module's `.codalotl/cas` root.
- Keep clarification behavior for dependency and stdlib packages, but do not attempt to write CAS outside the sandbox.
- Surface CAS write problems visibly rather than silently pretending the answer was recorded.

#### Package `internal/agentbuilder`
- Pass the current package-mode directory into `clarify_public_api` construction so CAS entries can record the package that originated the question.
- If the docs-improvement agent becomes unused after removing the automatic clarify path, remove or de-register it rather than preserving a dead legacy path.

#### Package `internal/noninteractive/integration`
- Update the `pm-clarify` replay/expectations to assert the new behavior: answer the question, do not patch source docs, and record clarify CAS metadata for the sandbox target.

### Review gates
- Run focused package tests for the changed packages and the `pm-clarify` integration case.
- When implementation is complete, run code review and changed-package SPEC conformance.

## Decisions

- Clarify CAS records are attached to the target package when that package is inside the active sandbox. Dependency and stdlib clarifications still answer, but do not write package CAS in this PR.
- A single CAS record per target package content should contain a list of clarification entries to avoid losing earlier questions for unchanged package contents.
