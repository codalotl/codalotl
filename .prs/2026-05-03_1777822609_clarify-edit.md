# PR

## User Summary (do not modify)

Background:
- we have the clarify_public_api tool, which answers a question about a package
- Because clarify was used at all, we might strongly suspect the package's public docs didn't naturally answer this question.
- It would be nice if the docs were more useful in the first place, resulting in faster and more reliable use of the package in question.

Solution:
- modify the clarify_public_api tool to (potentially) modify the docs
- the tool works in two phases:
    - the existing subagent works as it does now and does not change (it doesn't get apply_patch, for instance)
    - after that subagent ends, we spin up a new subagent, `improve_public_api_docs`
        - this subagent is only run if the package is inside the sandbox (clarifying a dep from the go.mod file does not edit that dep)
- This new subagent:
    - is jailed to the package
    - has typical package edit tools, but I suppose not change_api and update_usage and things like that
    - is given instructions to only improve docs, but only if it makes sense (like, no need to improve docs if the question is dumb)
        - I want this to be a markdown file in internal/agentbuilder/data
    - is given the question and the answer from the original clarify_public_api subagent
- UX:
    - user sees the clarify_public_api tool call, as they do today
    - user sees the stream of events from the clarify_public_api subagent, as today. The answer itself is hidden, as today
    - user sees more events under the clarify_public_api tool call (same nesting as the above subagent) for the improve subagent. For example, they'll see some Reads and Edits.
    - user sees a final message from the improve subagent; that's fine
    - user sees the clarify_public_api tool result, with the answer
    - In other words, the user mostly sees the same things, but with some extra events.

NOTE:
- The existing clarify_public_api tool is "read only".
- The new one is not. (But the clarify_public_api subagent is still read only).
- The user or caller does not need to know. They don't care. This is not a security or auth problem.
