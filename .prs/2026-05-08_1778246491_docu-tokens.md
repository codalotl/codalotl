## Plan

### Step 1: technical alternatives

Awaiting user choice among these alternatives.

#### Option A: agent-owned external LLM usage hook

Add an `agent` context helper for tool code to report token usage from LLM calls that happen outside the normal agent conversation. `docubot` would call that helper after each successful `llmstream.Completer.Complete` call, using the existing `BaseOptions.Context`.

- Shape:
  - `internal/agent`: add an exported no-op-safe helper such as `EmitExternalLLMUsage(ctx, modelID, usage)` and wire the tool-run context so it increments the owning agent's cumulative usage.
  - `internal/docubot`: after `Complete`, report the returned turn's `Usage`.
  - `internal/tui` / `internal/noninteractive`: mostly unchanged because their token displays already read `Agent.TokenUsage()`.
- Pros:
  - Correct architectural ownership: the agent owns usage accounting.
  - Covers both `codalotl_cli docs add` and `refactor docs-add`, because both ultimately pass the agent tool context into docubot.
  - Naturally covers future direct docubot calls inside agent tools.
- Cons:
  - Expands `internal/agent` public API for a non-conversation accounting path.
  - Needs care to not affect context-window usage; docubot tokens are cost tokens, not agent conversation context.

#### Option B: CLI-owned tracking completer

Keep `docubot` unchanged and have `codalotl docs add` pass a custom `llmstream.Completer` that wraps the default completer. The wrapper reports each returned turn's `Usage` to an agent context helper before returning the turn to docubot.

- Shape:
  - `internal/agent`: add a small no-op-safe external usage helper as in Option A.
  - `internal/cli`: add a tracking completer used only by `docs add`.
  - `internal/docubot`: no implementation changes.
- Pros:
  - Most targeted to the user-visible entrypoints: standalone `codalotl docs add`, `codalotl_cli`, and `refactor docs-add`.
  - Avoids coupling docubot itself to agent semantics.
  - Easier to test at the command boundary.
- Cons:
  - Future non-CLI docubot uses inside agent tools would need to remember to install the wrapper.
  - Usage reporting lives one layer away from the actual LLM call.

#### Option C: generic usage-reporting completer in `llmstream`

Add a reusable `llmstream` wrapper/decorator that invokes a callback after each successful completion. `codalotl docs add` installs that wrapper with a callback into `agent`.

- Shape:
  - `internal/llmstream`: add a `ReportingCompleter` / `CompleterFunc` helper.
  - `internal/agent`: add the external usage helper.
  - `internal/cli`: wrap the default completer for `docs add`.
- Pros:
  - Clean reusable primitive for any future one-shot LLM caller.
  - Keeps docubot decoupled from agent.
- Cons:
  - Slightly more abstraction than this PR needs.
  - Still requires callers to opt in to the wrapper.

#### Option D: parse/report docs command output

Have `docubot` print token/cost metadata to stdout, then teach `codalotl_cli` / `refactor` / TUI presentation to parse or surface it.

- Pros:
  - Avoids changes to `agent` internals.
- Cons:
  - Not recommended: brittle, duplicates token accounting/cost formatting, and does not update the existing TUI info-panel token totals naturally.

### Recommendation

Choose Option B unless you want docubot itself to always own reporting for all callers. Option B is the smallest change that covers both requested agent paths while preserving docubot's current API.

### Validation plan

- Unit tests around the external usage helper and the `docs add` tracking completer.
- Run focused package tests for changed packages.
- Run `check_spec_conformance({"only_changed":true})`.
- Manually validate with a judicious `go run . exec --yes ...` prompt that triggers `refactor` `docs-add` on a tiny package, then revert any generated documentation before committing.
