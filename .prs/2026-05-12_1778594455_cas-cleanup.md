## User Summary (do not modify)

I think there's some nonsense going on about CAS - multiple packages choose cas db locations in different ways. Lets unify. I documented in `product-spec`.

## Plan

### Phase 0: Centralize CAS DB root selection

#### Package `internal/gocas`
- Add the shared CAS root-selection API documented in `internal/gocas/SPEC.md`.
- Select root from `CODALOTL_CAS_DB` when set; otherwise use the nearest ancestor containing `.git` and root CAS at `<git-root>/.codalotl/cas`.
- Return absolute `BaseDir`/CAS root paths from constructors. Leave write authorization and directory creation to callers.

#### Package `internal/cli`
- Delegate CLI CAS root selection to `internal/gocas`.
- Preserve command side effects: write paths may create the CAS root; read-only/TUI paths should avoid creating it just by launching.

#### Package `internal/tools/refactor`
- Replace module-local `.codalotl/cas` construction with `internal/gocas` root selection.
- Authorize the selected CAS root through the tool authorizer; do not hard-code sandbox-local roots.

#### Package `internal/tools/spectools`
- Use `internal/gocas` root selection for retrieve/store/delete of conformance CAS records.
- Ensure CAS existence checks and write authorization use the same selected root.

#### Package `internal/tools/pkgtools`
- Use `internal/gocas` root selection for `clarify_public_api` CAS records.
- Update tests/fixtures that assumed module-local CAS roots.

#### Validation
- Run focused tests for touched packages.
- Run `go test ./...` if focused tests pass.
