## User Summary (do not modify)

Read:

- product-spec/SUMMARY.md
- product-spec/features/llms.md

In llms.md a two things are specified that are not implemented:
- If the subscription auth file is present but invalid or needs refresh or something, I think we possibly fall back to the API key. We need to not do that.
- The TUI needs to display that subscription auth is active for a model.

In this PR, we fix those.

NOTE: preferredprovider is not properly implemented. In this PR, we ignore it.

## Plan

### Phase 1: Subscription auth state and fallback suppression

#### Package internal/llmmodel [DONE]
- Add provider subscription-required state so provider-level API-key fallback can be suppressed when saved provider subscription auth exists but is unusable.
- Add a public helper for callers to ask whether a model is currently using provider subscription auth.
- Keep per-model explicit API key/env/endpoint overrides outside provider-level subscription suppression.

#### Package internal/subscriptions/openaisub [DONE]
- Mark OpenAI subscription auth required when default saved auth exists.
- Clear the requirement only when default saved auth is missing or logout removes it.
- Preserve current usable subscription registration behavior, but do not re-enable OpenAI API-key fallback merely because saved auth is expired, invalid, or refresh fails.

#### Package internal/cli [DONE]
- Ensure startup validation refreshes/checks saved OpenAI subscription auth before accepting OpenAI models via API key fallback.
- If the default subscription auth file exists but is unusable, fail with useful auth guidance rather than silently billing an API key.
- Ignore `preferredprovider` behavior in this PR.

#### Package internal/llmstream [DONE]
- Ensure OpenAI Responses request auth cannot fall back to provider-level API-key auth when llmmodel says provider subscription auth is required for that model.

### Phase 2: TUI subscription indication

#### Package internal/tui [DONE]
- Show that the active model is using subscription auth in the info panel.
- Annotate subscription-auth models in `/model` and `/models` output.

### Validation
- Run focused package tests for `internal/llmmodel`, `internal/subscriptions/openaisub`, `internal/cli`, `internal/llmstream`, and `internal/tui`.
- Run `go test ./...` if feasible.
- After implementation, run orchestrator review and changed-package SPEC conformance.

### Review follow-up [DONE]

#### Package internal/cli [DONE]
- Fix startup validation to only block a selected OpenAI model on unusable saved provider subscription auth when that model is eligible for provider subscription auth. Explicit per-model key/env/endpoint overrides should remain valid.

### Review follow-up 2 [DONE]

#### Package internal/cli [DONE]
- Fix startup validation so saved OpenAI subscription auth is refreshed before declaring that no models are available, even when the effective configured model is currently non-OpenAI and unavailable. Users with refreshable saved OpenAI subscription auth should not fail missing-auth startup validation merely because the configured effective model does not trigger the selected-model OpenAI refresh path.

### Review follow-up 3 [DONE]

#### Package internal/llmmodel [DONE]
- Fix provider-level API-key fallback suppression so an explicit per-model APIEnvKey only bypasses subscription-auth suppression when that environment variable has a usable value. A custom OpenAI model with an unset per-model env key must not fall through to provider-level `OPENAI_API_KEY` while saved OpenAI subscription auth is required but unusable.

### Review follow-up 4 [DONE]

#### Package internal/cli [DONE]
- Align startup subscription eligibility with `internal/llmmodel`: a custom OpenAI model with a configured but unset APIEnvKey should be treated as eligible for provider subscription auth. With unusable saved OpenAI subscription auth, startup should fail selected OpenAI models with the login/logout guidance even if another provider credential is available.

#### Package internal/llmstream [DONE]
- Align OpenAI Responses auth resolution with `internal/llmmodel`: a custom OpenAI model with a configured but unset APIEnvKey should be allowed to use usable provider subscription auth rather than skipping subscription auth solely because the env-var override name is present.

### Review follow-up 5 [DONE]

#### Package internal/cli [DONE]
- Ensure startup subscription refresh accounts for command-line model overrides. If a command selects an OpenAI model while config points to another authenticated provider, startup should refresh saved OpenAI subscription auth before request execution so refreshable saved auth does not remain required-but-unusable.
