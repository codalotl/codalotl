### External review: OpenAI ID token account claim

Accepted. Proof:
- Current code in `internal/subscriptions/openaisub/openaisub.go` rejects this case:
  - `Login` saves `tokens.ChatGPTAccountID`, then calls `authFile.normalized()`.
  - `authFile.normalized()` only fills `ChatGPTAccountID` from `accountIDFromJWT(auth.IDToken)` when the direct response field is empty.
  - `authFile.valid()` requires nonblank `ChatGPTAccountID`; otherwise `Login` returns `OpenAI subscription login did not return usable credentials`.
  - `accountIDFromJWT` currently reads only root `chatgpt_account_id` and plain `auth.chatgpt_account_id`, not `https://api.openai.com/auth.chatgpt_account_id`.
- Corroborating local OSS references:
  - `/home/jonathan/projects/others/codex/codex-rs/login/src/token_data.rs:65-91` models OpenAI ID tokens with `#[serde(rename = "https://api.openai.com/auth", default)] auth`, and maps `auth.chatgpt_account_id` at lines 135-142.
  - `/home/jonathan/projects/others/codex/codex-rs/login/tests/suite/device_code_login.rs:132-155` builds a successful device-login token using `"https://api.openai.com/auth": {"chatgpt_account_id": "acct_321"}` and expects the persisted account ID to be `acct_321`.
  - `/home/jonathan/projects/others/opencode/packages/opencode/src/plugin/codex.ts:48-72` types and extracts `claims["https://api.openai.com/auth"]?.chatgpt_account_id`.

Decision: fix `openaisub` to read the namespaced `https://api.openai.com/auth` claim, while preserving existing root and plain-`auth` fallback behavior.
