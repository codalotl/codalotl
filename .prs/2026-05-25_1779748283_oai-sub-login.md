### External review: OpenAI ID token account claim

Accepted. Proof:
- Current code in `internal/subscriptions/openaisub/openaisub.go` rejects this case:
  - `Login` saves `tokens.ChatGPTAccountID`, then calls `authFile.normalized()`.
  - `authFile.normalized()` only fills `ChatGPTAccountID` from `accountIDFromJWT(auth.IDToken)` when the direct response field is empty.
  - `authFile.valid()` requires nonblank `ChatGPTAccountID`; otherwise `Login` returns `OpenAI subscription login did not return usable credentials`.
  - `accountIDFromJWT` currently reads only root `chatgpt_account_id` and plain `auth.chatgpt_account_id`, not `https://api.openai.com/auth.chatgpt_account_id`.
- Corroborating local OSS references:
  - `/home/jonathan/projects/others/codex/codex-rs/login/src/token_data.rs:65-91` models OpenAI ID tokens with `#[serde(rename = "https://api.openai.com/auth", default)] auth`, and maps `auth.chatgpt_account_id` at lines 135-142.
  - `/home/jonathan/projects/others/codex/codex-rs/login/tests/suite/device_code_login.rs:132-155` builds a successful device-login token using `"https://api.openai.com/auth": {"chatgpt_account_id": "acct_321"}` and expects the persisted account ID to be `acct_321`.
  - `/home/jonathan/projects/others/opencode/packages/opencode/src/plugin/codex.ts:48-72` types and extracts `claims["https://api.openai.com/auth"]?.chatgpt_account_id`.

Decision: fix `openaisub` to read the namespaced `https://api.openai.com/auth` claim, while preserving existing root and plain-`auth` fallback behavior.

Fix implemented in commit `ddbdd5e`:

- Added focused coverage for extractor behavior, login persistence when the token response omits direct `chatgpt_account_id`, and refresh/status updating the saved account ID from a namespaced ID token.

### External review: missing token expiry fallback

Accepted. Proof:
- Current code in `internal/subscriptions/openaisub/openaisub.go` stores `ExpiresAt` as `now + tokens.expiresIn()` after both login (`openaisub.go:115-123`) and refresh (`openaisub.go:402-429`).
- `tokenResponse.expiresIn()` returns `defaultExpiresIn` when `expires_in <= 0`, and `defaultExpiresIn` is `7 * 24 * time.Hour` (`openaisub.go:27-39`, `openaisub.go:295-308`).
- `CheckStatusWithOptions` attempts refresh only when the saved auth is already expired and has a refresh token (`openaisub.go:152-174`), and `auth.valid` also treats the same future `ExpiresAt` as logged in (`openaisub.go:227-237`). Therefore, a token response that omits `expires_in` can make `auth openai status` skip refresh and report logged in until the artificial seven-day expiry even if the access token expires earlier.
- Existing package tests only exercise responses with `expires_in: 3600`; they do not protect the omitted-`expires_in` path.
- Corroborating local OSS references:
  - Codex Rust parses standard JWT `exp` via `parse_jwt_expiration` (`/home/jonathan/projects/others/codex/codex-rs/login/src/token_data.rs:93-127`) and uses the access token `exp` to decide whether ChatGPT auth is stale (`/home/jonathan/projects/others/codex/codex-rs/login/src/auth/manager.rs:1577-1597`).
  - Codex refresh tests demonstrate that a future access-token `exp` suppresses refresh and an expired access-token `exp` triggers refresh (`/home/jonathan/projects/others/codex/codex-rs/login/tests/suite/auth_refresh.rs:281-384`, helper at `1053-1055`).
  - opencode models `expires_in` as optional and uses `(tokens.expires_in ?? 3600)` in login/refresh persistence paths (`/home/jonathan/projects/others/opencode/packages/opencode/src/plugin/codex.ts:104-109`, `448`, `521`, `596`), a one-hour fallback rather than seven days.

Decision: fix `openaisub` to prefer explicit `expires_in`, then derive an absolute expiry from returned token JWT `exp` when available, otherwise use a conservative one-hour access-token fallback.

Fix implemented in commit `7c7002e`:
- Replaced the seven-day omitted-`expires_in` fallback with a one-hour fallback (`openaisub.go:27-39`).
- Login and refresh now both call shared token expiry calculation (`openaisub.go:115-124`, `openaisub.go:452-479`).
- Token expiry calculation preserves explicit positive `expires_in`, then reads access-token JWT `exp`, then ID-token JWT `exp`, then falls back to one hour (`openaisub.go:304-358`).
- Added focused coverage for access-token `exp`, ID-token secondary `exp`, explicit `expires_in` precedence, and fallback behavior (`openaisub_test.go:333-382`).
- Validation: `go test ./internal/subscriptions/openaisub`, `go run . spec diff internal/subscriptions/openaisub`, and `go test ./...`.

### User review: preserve `~/.codalotl` directory permissions

Accepted. The default auth file lives at `~/.codalotl/openai_auth.json`, but `~/.codalotl` is an app-wide directory that may contain user-editable or intentionally shared files. Saving OpenAI auth must not silently chmod an existing containing directory to `0700`.

Fix implemented in commit `5030614`:
- `saveAuth` still creates a missing containing directory with `0700`.
- `saveAuth` no longer chmods an existing containing directory.
- Auth file content remains written with enforced `0600`.
- Tests cover missing-dir private creation and preserving an existing `0755` directory while privatizing the auth file.

- Added Linux inotify coverage proving the chmod/attribute event occurs before any file write event during refresh.
