## User Summary (do not modify)

Improve the performance of `codalotl docs status`. Keep behavior the same. Only implement an optimization if it doesn't increase technical debt.

## Plan

### Phase 0: Measure before optimizing [DONE]

Measured `codalotl docs status` from a built binary so compile time is excluded:

- Run 1: real 24.23s, user 34.11s, sys 40.16s.
- Run 2: real 24.03s, user 33.99s, sys 40.47s.
- Run 3: real 24.09s, user 34.34s, sys 40.70s.

Measured `internal/cli` status phases across 71 packages:

- Package discovery: 0.23s.
- Package loading: 0.14s.
- `docs_add`: 17.07s.
- `docs_fix`: 0.02s.
- `reflow`: 5.73s.

Measured `internal/docubot` important-docs status path:

- Static important identifier selection: 1.56s.
- `gocodecontext.Groups`: 15.23s.
- Exact `CountMissingDocs` important count: 12.11s.

Conclusion: primary bottleneck is docs-add status, specifically exact important-docs counting and repeated graph grouping. Reflow is secondary. The previous reflow-first plan was wrong.

### Phase 1: Package `internal/docubot`

Add a boolean documentation-status API for `AddDocs` target selection, as described in `internal/docubot/SPEC.md`.

- Keep `CountMissingDocs` exact and preserve existing `AddDocs` target semantics.
- For `OnlyDocumentImportantIdentifiers`, answer "does this package need docs?" by checking cheap/static important targets first.
- Only run expensive `gocodecontext.Groups` work when static important targets are fully documented and graph-selected important identifiers can affect the boolean answer.
- Keep status checks read-only and LLM-free.

### Phase 2: Package `internal/cli`

Use the new `docubot` boolean status helper for the `docs_add` column.

- Preserve `codalotl docs status` output, package discovery behavior, status semantics, and read-only behavior.
- Keep `docs_fix` and `reflow` behavior unchanged in this phase.
- Remeasure after implementation using the same built-binary timing and phase breakdown.

### Phase 3: Optional reflow follow-up

Only optimize reflow if post-Phase-2 measurements still show it is material and the fix is clean.

- Likely owner is `internal/cli` if the right fix is batching/shared package clones.
- Consider `internal/updatedocs` only if a better dry-run status API is warranted.
- Do not implement a reflow optimization speculatively.
