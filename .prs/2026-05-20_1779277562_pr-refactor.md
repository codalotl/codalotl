## User Summary (do not modify)

see product-spec/features/pr-orchestrator.md

In this per, we build `codalotl pr refactor --package=<path/to/pkg>`.

Order of refactors:
- docs add
- docs fix
- dry
- test cleanup
- test coverage

Example instructions (this will show up in the User Summary section of PR file):

```markdown
In this PR, we refactor path/to/pkg.

During implementation, apply the following refactors:
- refactor("name": "docs-add", "package": "path/to/pkg")
...
- refactor("name": "test-ensure-coverage", "package": "path/to/pkg")

Additional instructions:
- After each refactor, inspect diff.
    - If diff looks good, add and commit, including CAS file.
    - If diff looks problematic, either reset and try again (max 1 retry), spawn an `implement` agent to fix forward, or make a note in PR file and skip. What you choose to do is based on what problem looks like. Important: these are intended to be safe, low risk refactors. Do NOT add risk by making large changes. These refactors are not intended to change the public API or behavior of a package (except for adding/changing docs). They are not intended to fix production bugs.
- After the last refactor is commited, use the `codalotl_cli` tool to run `codalotl cas recertify --namespaces="docs-add,docs-fix,..." path/to/pkg`. Commit CAS files. This marks the last version of the the package as compliant with all refactors.
- Do the above implementation steps in one turn (all refactors). Combine it with your planning step, since this is effectively the plan.
- Beyond the Plan and Implement steps, the rest of the workflow (Review, etc) remains the same.
```
