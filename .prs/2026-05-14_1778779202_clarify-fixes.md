# PR

## User Summary (do not modify)

We recently let clarify_public_api make further clarify calls. This was a big mistake, leading to massive clarify spidering over the codebase clarifying everything, including its caller, creating potential clarify cycles.

To fix:
- remove clarify_public_api as a tool inside clarify_public_api.

moreover, let's clarify the language in the clarify instructions (pun intended):
- include language that get_public_api should only be used on deps that exist in the clarified package, not on various other packages in the codebase.
- include language that clarify should just answer the best it can by looking at its own code and its dep's public api
    - If that's not enough to answer the question, that's ok! don't guess - just state the facts you know and what you don't know. Examples:
        - a package is sometimes a "types-only" package. answering questions about it is more a question for other packages - the ones that use that type. this shows up for interface types, as well as structs.
        - likewise, any time deps are "inverted" in some way, the package won't adequately be able to answer specifics in a way that is satisfactory to the caller.
        - in cases like the above, clarify should essentially say, "bro i'm just a type".
        - another case where clarify should say what it knows and what it doesn't:
            - when the pckage calls into another package, but that package's public api isn't adequately documeneted.
            - so clarify can report something like, "bro, the function you inquired about calls Foobar. Foobar does X, but i'm not able to determine Y. What I can say is Z."

For the above prompt changes: do a really good job. I typed out the direction, but obviously you gotta think hard on this to optimize it.
- Come up with 3 different versions. Use `implement` to spin up another agent somehow with fresh context to vote on the best one and give you feedback. Maybe via a new tmp package that you write.

Validation: in addition to normal go test, manually validate this works by running something like `go run . exec` that invokes clarify to ask a question of some package, where clarify would otherwise want to recursively clarify. Maybe clarify something about internal/tools/toolsetinterface. Observe output; be on the lookout for ux regressions.
