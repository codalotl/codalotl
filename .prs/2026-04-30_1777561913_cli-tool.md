# PR

## User Summary (do not modify)

Problem:

- codalotl has many useful CLI commands. Ex: `codalotl context initial`, `codalotl docs add path/to/pkg`, etc.
- in TUI mode, i want to tell the agent to "run codalotl's xyz command"
    - (but this also applies to noninteractive mode)
- But locally, the codalotl binary may not have been installed. We might be in "go run" mode, or in a test.

Solution:

- Make a tool `codalotl_cli`. Params, both required:
    - `subcommand`: if `codalotl` is the command, then the subcommand(s) are the things like `context initial` or `docs add`. Doesn't always have to be 2 words. Can be 1 or 3 or ...
    - `argv`: []string - the arguments to the `codalotl` subcommands. Ex: ['internal/mypkg'] or ['--public-only', 'internal/mypkg'] or ['--search=abcd'] or [] or null.
        - TODO: is this better as args string or argv []string?
        - Instructions for LLM: decide on string vs []string, picking best one.
- I think this tool can live in: `internal/tools/cli` (arguments can be made for various other places)
- So `internal/cli` creates this tool, adds some of its commands to it, and adds it to the registry.
- Add this tool to the generic agent.

This tool itself does NOT know anything about `internal/cli` or which subcommands there are. It is constructed with some `New` type function with a mapping of some kind, or a way to add commands to the resultant tool.

Things I want in the design:
- The tool itself (from the LLM perspective) should be self describing:
    - `codalotl_cli({"subcommand": "help"})` AND `codalotl_cli({"subcommand": "--help"})` should list available commands. `--help` is mentioned in the tool's description.
    - This command also explains that any subcommand can accept `--help` for more details. But --help is never listed as an option in the command's own oneliner or help section.
    - In the actual `codalotl` command, `codalotl --help` only lists root nodes, like `codalotl docs`. But here, it should be expanded, listing only leaves, one per line. 
        - Ex (exact format not prescriptive): `codalotl docs add [--public-api] [--include-test] <pkg>    Add missing documentation comments to a package.`
    - The oneliner:
        - always prepends `codalotl`.
        - then the command
        - then either `[OPTIONS]` if there's many options (if there's none (other than --help): don't incldue it)
        - or [--some-option] [--other=option=<VAL>] if there's 3 or fewer option
        - then non-option args.
            - Use `<pkg>` for a mandatory packge. Use `<something>` for a mandatory something.
            - Use `[<other-thing>]` for an optional other thing.
        - Then a short description.
    - Running `codalotl docs add --help`:
        - Longer description.
        - Prints all options with documentation for each (usually a sentence or sentence fragment)
        - Prints all args iwth documentation for each.
        - Prints examples

Example:

```text
$ codalotl docs add --help
Add missing documentation comments to a package. Never edits comments. Ensures all package-level identifiers have godoc-style doc comments.

Options:
    --include-test  Include test files (by default, test files are not documented).
    --public-only   Only document exported identifiers.

Args:
    <pkg>   path/to/pkg (relative to sandbox dir) or Go import path of package to document.

Examples:
    # Add docs to public/private code in internal/mypkg (excludes test files):
    codalotl docs add internal/mypkg

    # Add docs to public code in internal/mypkg (excludes test files):
    codalotl docs add --public-only internal/mypkg
```

Things I want in the design (cont):
- This should be roughly aligned with `internal/cli` in terms of the help content, args handling, etc. I gave some rather specific things in the design of --help. These don't currently align exactly with cli.
    - So first, align q/cli with this direction (if necessary).
    - Then express existing CLI with that. Right now the CLI help text is horrible.
    - Then build internal/tools/cli
- internal/tools/cli MAY have a dependency on q/cli. It's not necessary, but it is allowed. If it's helpful, we can do that.
- It should be extremely easy to add an existing or new CLI command to codalotl_cli tool. Not all cli commands belong in codalotl_cli! They are whitelisted in.
    - I'm not sure if this is necessary. But i think I want the **option** to customize things like help text in cli vs codalotl_cli. For instance, a llm-oriented help text might have guidance like "ONLY use this if you can't use your module_info tool", which makes no sense for a human.

Ok, more things:
- In order to codalotl_cli-ize a command, the command MUST support a parametric Stdout (io.Writer). The tool will create a io.Writer of some kind, which the command will write to.

So that's interesting. Typically this type of display is for subagents, implemented with `internal/agent` and `internal/agentregistry` and all that. But here, these sub-bullets are actually just lines printed by the io.Writer in the CLI code. So we need to support that somehow. I think the main primitive needed is the ability to trigger synthetic agent messages. Then:
- When do we actually do the call to in the tool to trigger the synthetic agent message?
    - On newlines? What if a bunch of output but no newlines?
    - What the CLi code does somthing like this: Fprintf(w, "data:\n- field1: %s\n- field2: %s\n", f1, f2)
        - I think this is better if the User sees this with one bullet/message.
    - So. You come up with specific algo. Roughly:
        - debounce on newline. Flush after, say, 1000ms.
        - flush of course if the cli call returns, even if no newline.
- These bytes need to be sanitized:
    - no tabs
    - allow ansi control? not sure.
    - trim line length. Ex: limit to, say, 400 characters.
    - If the command absolutely SPEWS data (lots of lines, really fast):
        - in the debouncing/flushing code, can elide lines with "... (239 more lines)"
    - what else?
- Generally, the printing of the stuff for the user is a nice-to-have and can be more aggressively sanitized/limited.
- The tool returns the bytes in a more complete manner to the LLM. can still be sanitized a bit (see read file). But don't, for instance, do aggressive "... (239 more lines)" eliding.
- Tool result's presenter doesn't need to re-display the otuput, since we already printed it as synthetic messages.
