# PR

## User Summary (do not modify)

In the past few PRs, we added a way to log in with chatgpt credentials, which is what all OSS agents do. Currently, this just saves credentials in a file. They're accessed via the internal/subscriptions/openaisub package. We also added support for things we need: zdr-style response api usage (NoStore) and using instructions.

In this PR, we update llmstream to use the openai subscription if it exists.

But first, we need to sort out deps. I do not want llmstream to depend on openaisub - it's too high level and too weighty. Instead, upon init, openaisub "registers" some sort of subscription credentials into llmmodel. llmstream can then access that via llmmodel. llmmodel has a public api directionally aligned with this (you can modify this API if necessary.):

```go
// ProviderSubscription is provider-agnostic subscription auth that can be used instead of a provider API key.
type ProviderSubscription struct {
	ProviderID       ProviderID
	AccessToken      string
	AccountID        string
	APIEndpointURL   string
	ExpiresAt        time.Time
	RequiresNoStore  bool
	RootInstructions bool
}

// SetProviderSubscription configures subscription auth for a provider.
func SetProviderSubscription(providerID ProviderID, sub ProviderSubscription)

// ClearProviderSubscription removes subscription auth for a provider.
func ClearProviderSubscription(providerID ProviderID)

// GetProviderSubscription returns subscription auth for providerID, if set.
func GetProviderSubscription(providerID ProviderID) (ProviderSubscription, bool)

// ProviderHasSubscription reports whether subscription auth is configured for providerID.
func ProviderHasSubscription(providerID ProviderID) bool
```

llmstream will query for a subscription. If will only use it if:
- provider matches and is "logged in"
- no overriden URL
- no overriden api key env var
- (we still use the subscription over the api version even if the api key is set)
- Is there anything else? Please comment if this feels wrong or needs amends.

## Plan

### Phase 0: provider subscription registry [DONE]

#### Package `internal/llmmodel` [DONE]
- Add provider-agnostic subscription registry API described in `internal/llmmodel/SPEC.md`.
- Treat subscription auth as usable only when it has provider match, nonblank access token/account ID/endpoint, and unexpired credentials.
- Add `AvailableModelIDsWithAuth()` for callsites that should accept either API-key auth or eligible subscription auth.
- Subscription eligibility is provider-level but suppressed by per-model custom API key/env or endpoint overrides. Default provider API keys/env vars do not suppress subscription auth.

### Phase 1: OpenAI subscription registration [DONE]

#### Package `internal/subscriptions/openaisub` [DONE]
- Register usable OpenAI ChatGPT subscription credentials with `llmmodel` on package init/default saved auth load, login success, and status refresh/load.
- Clear registered OpenAI subscription auth on logout or unusable status.
- Registered subscription should target ChatGPT Codex-compatible Responses, include account ID, require no-store behavior, and support root instructions.

### Phase 2: OpenAI Responses streaming [DONE]

#### Package `internal/llmstream` [DONE]
- Resolve OpenAI request auth through `llmmodel`: use eligible subscription auth when present; otherwise use existing API-key behavior.
- For subscription requests, send bearer/API-key auth, `ChatGPT-Account-ID`, and subscription endpoint URL.
- Force effective NoStore behavior for subscription auth that requires no-store; this must affect request params, response-linking, retained turns, and replay.
- Make HTTP request debug logging include the resolved request path so manual validation can prove the ChatGPT Codex endpoint is being hit.
- Ensure final assistant turns are derived from streamed state when the completed response output array is empty.

### Phase 3: downstream availability and validation

#### Package `internal/cli` [DONE]
- Startup validation accepts usable provider subscription auth as LLM configuration.
- Missing-auth guidance describes both provider API keys and supported provider subscription login.

#### Package `internal/tui` and model-selection callsites [DONE]
- Model-selection surfaces use auth-aware availability where appropriate, so subscription-only OpenAI auth can run TUI flows with no OpenAI API key.
- Empty-state guidance mentions provider API keys and supported provider subscription login.

## Review

External automated review feedback assessed before formal review:
- [DONE] Expired saved OpenAI subscription access tokens with valid refresh tokens are not refreshed during default startup load. `openaisub` init loads the default auth file and immediately syncs it into `llmmodel`; expired auth fails eligibility and clears subscription auth. Since startup validation now uses `llmmodel.AvailableModelIDsWithAuth()` instead of `openaisub.CheckStatusWithOptions()`, subscription-only users can be rejected after access-token expiry until they run an explicit auth status/login command.
- Fix implemented in `internal/subscriptions/openaisub`: default startup load refreshes expired default saved auth with a bounded context before syncing provider subscription auth. Refresh failure preserves the saved auth file but clears unusable registered subscription auth.

Formal review against `main`:
- [ACTIONABLE] P2: startup refresh currently happens from `openaisub` package initialization. That can block unrelated commands such as help/version/logout before argument parsing or startup-validation opt-out logic. Fix by moving network refresh to an explicit startup path that runs only when subscription-backed model availability is needed.

## Decisions

- `llmstream` will not import `openaisub`; `openaisub` registers provider-agnostic subscription auth in `llmmodel`.
- Per-model/custom API key env, actual key, or endpoint URL override suppresses subscription auth. Default provider API key/env does not.
- Subscription-only auth should count as configured for startup/model availability.
- OpenAI ChatGPT subscription requests should use the ChatGPT Codex-compatible Responses endpoint and `ChatGPT-Account-ID` header.
