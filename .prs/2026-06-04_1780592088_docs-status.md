# PR

## User Summary (do not modify)

Add `codalotl docs status`.

Goal: give the user a repo/package overview of documentation work remaining, similar in spirit to `codalotl spec status`.

Suggested behavior:
- Discover packages the same way `codalotl spec status` does: across modules discovered from the nearest git repo root.
- Print a compact human-readable table, one package per row.
- Include status for:
  - `docs add`: computed from the current package source, not CAS. Do not add docs-add CAS. Prefer the same missing-doc target set used by the `docs-add` refactor (`docs add --important`) unless implementation review finds a better user-facing default.
  - `docs fix`: CAS-backed status from the whole-package `docs-fix` namespace. Identifier-limited docs-fix records should not make a package look fully current.
  - `docs reflow`: include a deterministic check status if it can be done cleanly and reasonably cheaply. If this gets awkward, keep the first implementation focused on docs-add/docs-fix and leave reflow as a clear follow-up.
- Exact column names/status labels are not prescribed. Prefer terse values like current/needed/stale/unset/error over verbose explanations.
- The command should be useful from the CLI and, if consistent with the existing whitelist pattern, from `codalotl_cli`.

Validation:
- in addition to normal validation,
- also try to actually run this (~ `go run . docs status`). For the result, hand-check the result to verify it's correct for this repo.
