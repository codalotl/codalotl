# PR

## User Summary (do not modify)

Problem:
- In internal/tui and internal/noninteractive, there are features that rely on subagent final-message handling - notably, formatters for the final message (sometimes JSON, which needs to be parsed; sometimes hidden)
    - See llmstream.SubagentFinalMessagePresenter
- Determining the final message is annoying and error-prone. Since there are multiple clients (tui vs noninteractive), the functionality is duplicated.
    - One annoyance: assistant text messages come in parts. Adjacent parts may (in theory) need to be combined to form one message
    - Also, there can be multiple messages per turn: an assistant text, followed by a thinking blob, then another assistant text, etc.
    - This is doubly annoying because **in practice**, none of this actually happens. Messages are not split. But according to the API docs' object model, it "could". And it's super rare for there to be more than one assistant message in a turn.

Clarification of "final message":
- I am not sure what the current code's definition is. However, our definition is more like "finalizing message" -- the message at the end of the conversation. The reason is that this is used for things like "The Agent's Final Answer" or "The Agent's JSON message after it did a bunch of stuff". If it emits a message, then reasoning, then abruptly ends the convo with DoneSuccess, that is NOT a final answer. Again: it's not the most recent agent text event. It only counts as "final message" if there's nothing after it.

In this PR:
- Move final message handling to agent
- Use this in tui/noninteractive, removing the messy code from there (as it relates to figuring out which message is 'final').
- Make the agent mental model: an assistant text is NOT a part - it's a full message
- The agent will need to hold on to assistant text events until the next event comes in

Tension:
There is a fundamental tension between streaming live events from llmstream -> agent -> {consumer} and the final canonical Turn in DoneSuccess. In theory (but not in practice!), these can disagree.

Normal/Good case (streamed agrees with final turn):

```
  text_delta("beginning work")
  reasoning_delta("thinking")
  text_delta("done with work")
  completed_success(turn = [Text("beginning work"), Reasoning("thinking"), Text("done with work")])
```

Bad case:
```
  reasoning_delta("thinking")
  tool_use(call_1)
  completed_success(turn = [Text("Hello"), Reasoning("thinking"), ToolCall(call_1)])
```

Bad case 2:
```
  text_delta("answer")
  completed_success(turn = [Text("draft "), Text("answer")])
```

Our policy:
- non-assistant-text events must be emitted as they arrive. Reason: the user wants to see things as they happen, esp. in very long-running convos.
- Usually (except for the case below), we do NOT reconcile events and completed turns. If they disagree, that sucks, but we're not going to care much.
- If we're holding only assistant text events, and a completed turn comes in which disagrees with events in some way (see bad case 2), then the completed turn just wins.
    - So in "Bad case 2", the agent emits a single assistant text event of "draft answer", which is marked as the final message.

Concrete events emitted in "Bad case 2":
  - assistant_text("draft answer", finalizing=true) or equivalent final-message metadata
  - assistant_turn_complete
  - done_success

## Design

This should stay primarily within `internal/agent`, `internal/noninteractive`, and `internal/tui`.

### Package `internal/agent`

- Change the semantics of `EventTypeAssistantText` from "a completed llmstream text part" to "a logical assistant text message within one assistant turn".
- Add final-message metadata on assistant-text events so downstream consumers can tell whether a given assistant text is the finalizing message for that agent turn.
- In `sendOnce`, buffer completed assistant text until either:
  - a non-assistant-text event arrives, in which case flush the buffered text as a non-final assistant-text event before emitting that next event; or
  - `llmstream.EventTypeCompletedSuccess` arrives, in which case reconcile the buffered text against the end of the completed turn and emit exactly one finalizing assistant-text event if the completed turn ends in assistant text.
- Coalesce adjacent streamed assistant-text fragments into one logical message so consumers do not need to reconstruct message boundaries themselves.
- Keep `EventTypeAssistantTurnComplete` and the completed `Turn` as the canonical history/reconciliation source. The new assistant-text metadata is for streaming/presentation, not a replacement for the completed turn.
- Update `collect_answer.go`, `internal/agent/SPEC.md`, and focused tests to cover:
  - assistant text followed by reasoning
  - assistant text followed by tool use
  - turns ending in assistant text
  - turns ending without assistant text
  - buffered-text disagreement where the completed turn wins
  - descendant/subagent streams

### Package `internal/noninteractive`

- Remove the descendant final-message buffering/reconstruction logic from `subagentDisplayFilter`.
- Keep only the tool/subagent scope tracking needed to associate a descendant finalizing assistant-text event with the launching tool presenter.
- When a descendant finalizing assistant-text event arrives and the launching tool presenter implements `llmstream.SubagentFinalMessagePresenter`, replace or suppress that one event using the presenter. Non-final descendant assistant text should continue to flow through immediately.
- Simplify top-level `Result.FinalAssistantText` handling to use the top-level finalizing assistant-text event rather than rebuilding it from partial text plus `EventTypeAssistantTurnComplete`, while still storing completed turns for token/caching reporting.
- Update `internal/noninteractive/SPEC.md`, unit tests, and replay/integration fixtures if the event ordering changes.

### Package `internal/tui`

- Remove the per-tool pending descendant-turn buffering (`pendingDescendantTurn`, `pendingDescendantTurnComplete`, and related flush/finalize logic).
- Keep the active tool-scope and subagent-label bookkeeping, because the TUI still needs to know which tool presenter applies to a given descendant finalizing message.
- When a descendant finalizing assistant-text event arrives, display it as plain text by default, or route it through `llmstream.SubagentFinalMessagePresenter` to hide/replace it.
- Leave intermediate descendant assistant text live in the stream; only the finalizing message gets special treatment.
- Update `internal/tui/SPEC.md` and focused tests for the new event semantics.

### Scope Notes

- No `internal/llmstream` API change is required for this PR. `llmstream.SubagentFinalMessagePresenter` can remain a presentation hook; `internal/agent` only takes over message-boundary/finality detection.
- This refactor should remove the duplicated "is this the final descendant message?" buffering logic from TUI/noninteractive, but it will not eliminate all consumer-side subagent state. The consumers still need enough scope-tracking to map descendant finalizing messages back to the tool call that launched them.
