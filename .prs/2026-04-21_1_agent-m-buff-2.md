# PR

## User Summary (do not modify)

Problem:
- In internal/tui and internal/noninteractive, there are features that rely on subagent final-message handling - notably, formatters for the final message (sometimes JSON, which needs to be parsed; sometimes hidden)
    - See llmstream.SubagentFinalMessagePresenter
- Determining the final message is annoying and error-prone. Since there are multiple clients (tui vs noninteractive), the functionality is duplicated.
    - One annoyance: assistant text messages come in parts. Adjacent parts may (in theory) need to be combined to form one message
    - Also, there can be multiple messages per turn: an assistant text, followed by a thinking blob, then another assistant text, etc.
    - This is doubly annoying because **in practice**, none of this actually happens. Messages are not split. But according to the API docs' object model, it "could". And it's super rare for there to be more than one assistant message in a turn.

Clarification of "final message":
- I am not sure what the current code's definition is. However, our definition is more like "finalizing message" -- the message at the end of the conversation. The reason is that this is used for things like "The Agent's Final Answer" or "The Agent's JSON message after it did a bunch of stuff". If it emits a message, then reasoning, then abruptly ends the convo with DoneSuccess, that is NOT a final answer. Again: it's not the most recent agent text event. It only counts as "final message" if there's nothing after it.

In this PR:
- Move final message handling to agent
- Use this in tui/noninteractive, removing the messy code from there (as it relates to figuring out which message is 'final').
- Make the agent mental model: an assistant text is NOT a part - it's a full message
- The agent will need to hold on to assistant text events until the next event comes in

Tension:
There is a fundamental tension between streaming live events from llmstream -> agent -> {consumer} and the final canonical Turn in DoneSuccess. In theory (but not in practice!), these can disagree.

Normal/Good case (streamed agrees with final turn):

```
  text_delta("beginning work")
  reasoning_delta("thinking")
  text_delta("done with work")
  completed_success(turn = [Text("beginning work"), Reasoning("thinking"), Text("done with work")])
```

Bad case:
```
  reasoning_delta("thinking")
  tool_use(call_1)
  completed_success(turn = [Text("Hello"), Reasoning("thinking"), ToolCall(call_1)])
```

Bad case 2:
```
  text_delta("answer")
  completed_success(turn = [Text("draft "), Text("answer")])
```

Our policy:
- non-assistant-text events must be emitted as they arrive. Reason: the user wants to see things as they happen, esp. in very long-running convos.
- Usually (except for the case below), we do NOT reconcile events and completed turns. If they disagree, that sucks, but we're not going to care much.
- If we're holding only assistant text events, and a completed turn comes in which disagrees with events in some way (see bad case 2), then the completed turn just wins.
    - So in "Bad case 2", the agent emits a single assistant text event of "draft answer", which is marked as the final message.

Concrete events emitted in "Bad case 2":
  - assistant_text("draft answer", finalizing=true) or equivalent final-message metadata
  - assistant_turn_complete
  - done_success

## Plan

The original design is directionally right, but too broad in a few places:
- it redefines `assistant_text` without pinning down the exact event/API change consumers will read
- it implies `internal/tui` and `internal/noninteractive` can drop most subagent state, but both still need tool-scope and label tracking to map descendant messages back to the launching tool presenter
- it does not clearly separate internal agent metadata from `internal/noninteractive`'s public JSON log schema

This replacement plan keeps the same user-visible goal, but narrows it to concrete package changes.

### Phase 0

In this phase, move final-message detection into `internal/agent` and make consumers consume that signal directly.

#### Package `internal/agent`

- Extend `agent.Event` so `EventTypeAssistantText` can indicate whether the emitted logical message is finalizing for that agent's current provider send. Keep `EventTypeAssistantTurnComplete` as the canonical completed-turn/history event.
- Tighten the semantics of `EventTypeAssistantText`: it should represent one logical assistant text message, not an arbitrary completed `llmstream` text block.
- Implement this in `sendOnce` with buffering scoped to one provider send for one agent:
  - buffer consecutive completed assistant-text blocks
  - flush them as one non-final assistant-text event before any non-text event
  - on `llmstream.EventTypeCompletedSuccess`, inspect only the trailing assistant-text run of the completed turn
  - if the completed turn ends in assistant text, emit exactly one finalizing assistant-text event; when buffered streamed text disagrees with the trailing completed-turn text, the completed turn wins
  - if the completed turn does not end in assistant text, drop any still-buffered text rather than mislabeling it as final
- Update `collect_answer.go` to prefer the new finalizing assistant-text signal for the target agent instead of reconstructing from arbitrary text accumulation; keep terminal/error behavior intact.
- Update `internal/agent/SPEC.md` and focused tests for:
  - assistant text followed by reasoning
  - assistant text followed by tool use
  - turns ending in assistant text vs non-text
  - adjacent trailing text parts merged into one finalizing event
  - completed-turn disagreement where the completed turn wins
  - descendant/subagent streams
- Review/conformance follow-up [DONE]:
  - concatenate adjacent text parts without paragraph separators
  - flush already-complete buffered text before cancel/error/abort exits
  - on `CompletedSuccess`, still emit non-final assistant text when the completed turn contains earlier text followed by reasoning/tool content that was not surfaced as intermediate events
- Retry follow-up:
  - do not let assistant-text runs flushed before `EventTypeRetry` cause the eventual successful completed turn to skip its own text runs

#### Package `internal/noninteractive` [DONE]

- Keep only the subagent scope bookkeeping needed to map a descendant event back to the launching tool presenter (`activeTools`, `StartSubagent`, label/scope tracking). Remove pending descendant assistant-turn reconstruction.
- Treat non-final descendant assistant-text events like ordinary streamed text.
- When a descendant finalizing assistant-text event arrives and the launching tool presenter implements `llmstream.SubagentFinalMessagePresenter`, replace or suppress only that one event.
- Set `Result.FinalAssistantText` from the top-level finalizing assistant-text event only. Keep `EventTypeAssistantTurnComplete` storage for token/caching reporting, not final-message detection.
- Keep the noninteractive JSON output schema unchanged unless implementation proves an internal-only flag is required there. If event timing/order changes, update replay/integration fixtures intentionally.
- Update `internal/noninteractive/SPEC.md` and focused tests for both human-readable and JSON-output modes.

#### Package `internal/tui` [DONE]

- Keep ancestor/tool-scope tracking and subagent labels so descendant finalizing messages can still be associated with the launching tool presenter.
- Remove per-tool pending descendant-turn buffering and `EventTypeAssistantTurnComplete`-based descendant final-message detection.
- Let non-final descendant assistant text flow immediately into the transcript.
- When a descendant finalizing assistant-text event arrives, either display it plainly or route it through `llmstream.SubagentFinalMessagePresenter`.
- Update `internal/tui/SPEC.md` and focused tests for the new event semantics.

### Validation

- Run focused package tests for `internal/agent`, `internal/noninteractive`, and `internal/tui`.
- Update replay/integration fixtures only where event ordering or assistant-text grouping intentionally changed.
- After implementation, run review and changed-package SPEC conformance.

## Decisions

- Keep `EventTypeAssistantTurnComplete` as the source of record for history and usage. The new assistant-text finalizing flag is a streaming/presentation signal, not a replacement for completed turns.
- Do not change `internal/llmstream` for this PR.
- Do not expand the noninteractive JSON log schema unless implementation reveals a concrete external need.

## Learnings

- Rejected one attempted fix for `internal/noninteractive/integration` conformance: adding `hello-world`-specific fallback behavior in package code. The spec issue should be fixed in case data (or, if truly warranted, in the spec), not by teaching the harness about one special case name.

## Review

Reviewed against `main`.

- Actionable review findings in `internal/agent`:
  - `coalesceTextContent` currently uses `llmstream.Turn.TextContent()`, which inserts `\n\n` between adjacent text parts and can corrupt raw final messages such as JSON.
  - buffered completed assistant text can be dropped on cancel/error/missing-`completed_success` exits instead of being shown as non-final streamed text.
- Changed-package SPEC conformance:
  - `internal/agent`: non-conforming (major). `sendOnce` can drop non-final assistant text when `CompletedSuccess` contains earlier text followed by reasoning/tool content that did not appear as intermediate events.
  - `internal/noninteractive`: conforms.
  - `internal/tui`: conforms.
  - `internal/noninteractive/integration`: non-conforming (trivial, latent). Its `SPEC.md` says `hello-world` is a self-contained case, but `testdata/cases/hello-world` currently relies on the shared repo fixture.
- Follow-up status:
  - `internal/agent` fixes landed for raw concatenation, abort flushing, and completed-turn recovery of missing non-final text.
  - `internal/noninteractive/integration` conformance cleanup landed via case data; `hello-world` is now self-contained.
  - Rerun result: one remaining `internal/agent` retry bug; changed-package SPEC conformance now only fails for that same `internal/agent` issue.
- Rerun review against `main`:
  - actionable: if `llmstream` emits `EventTypeRetry` after completed text blocks, `sendOnce` can incorrectly count those pre-retry text runs against the later successful completed turn and drop the real final text.
- Rerun changed-package SPEC conformance:
  - `internal/agent`: still non-conforming (major) for the same retry scenario.
  - `internal/noninteractive`, `internal/noninteractive/integration`, `internal/tui`: conform.
