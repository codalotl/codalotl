# PR

## User Summary

Background/Problem:

When in orchestrator mode (or generic mode), I want to be able to tell an agent to apply various canned refactors. Examples:
- use/combine shared helpers in a package.
- refactor tests
- find documentation errors
- reorganize files in a package
- add docs in a package

I suspect the user may want to define custom, per-project refactors, only relevant for them/their company/their sensibilities.

I think all refactors will be confined to a package. There can definitely be cross-package refactors, but that's out of scope for this. I want to make per-package refactors really good.

I think ~most of these refactors will rely on "CAS" to store results based on a hash of package/code unit/etc.
- The fundamental thing there: is it necessary to run, say, "reorganize files in a package" refactor?
- I believe CAS can be extended (in the future) to be usable for that here. (we may need, say, for package foo, how long ago did we do the thing? what is the diff since then?)

I think some of these refactors will be agent-based. Essentially, just a custom prompt in package mode. "clean up tests. guidelines: ...". However, some of these will involve custom logic:
- find doc errors uses docubot to efficiently scan tons of docs quickly.
- reorganize files may similarly use LLM to make a target map, then use a single code transform to move functions around.

We already have a menagerie of abstractions that could potentially do something like this:
- Each refactor could be its own tool
- build "spawn_subagent" tool - some refactors like the custom prompt test cleanup could use this.
- put some/all refactors as CLI commands in codalotl_cli
- Maybe just document some of these refactors as skill files.
- Maybe the `implement` tool/subagent just has access to codalotl_cli, and a skill file on how to use it, to do refactors. Maybe it requires a flag set on calling it (implement(refactor: true, "do xyz refactor..."))
- some combination thereof (e.g., skill file can reference codalotl_cli, which exposes things like `docs fix` and `cas` stuff)

Of those:
- a tool per refactor is out (too many tools)
- spawn_subagent is out (doesn't solve all cases, and not necessary. Full solution here is nice but complex - see Codex's subagent stuff; mailboxes; etc)
- all refactors as CLI commands is out (although not terrible, feels a bit off, esp. for refactors that are just prompt+normal agent). The implementation for those would essentially just be "exec".

I think there are 2 viable solutions:
- somehow teaching `implement` about how to do refactors. Maybe conditionally give it access to refactoring skills and codalotl_cli. It manages CAS by running codalotl_cli, based on a skill workflow. One nice property: if a non-agent transform fails, the agent can fix it. Downside: REQUIRES spending more tokens on agent stuff to drive everything.
- implementing a single `refactor` tool. Downside: another tool. Upside: it can do exactly what we want, and doesn't need to be driven by LLM.

Solution:

We're going to implement a `refactor` tool. For now, the `generic` agent will have access (as we try it out and gain experience). Future PRs will enable it for, say, orchestrator.

Package: `internal/tools/refactor`

Tool Shape:

refactor({
    "name": "docs-add",             // enum of refactors available
    "package": "internal/foo",      // refactor is always package-specific (import path or relative dir). Supplied package must be in sandbox; if the package is outside the sandbox, return an error (e.g., "package": "fmt" or "package": "github.com/x/y", where x/y is some go.mod dependency)
})

To start, these refactors will be available. Each refactor is listed in the tool's description (as seen by the LLM). This list is dynamic based on implemented refactors.
- docs-add: this just calls the same thing that `codalotl docs add --public-only` does.
- dry: this refactor DRYs up a package - it looks for missing helper functions, creates them, and then uses them. It also sees if it can combine similar-or-identical helper functions into one version.
    - This refactor is "new" - it doesn't exist in this repo yet.
    - This is just a "prompt"-style refactor.
    - Uses the "limited_package_mode" agent

CAS:
- "docs-add" does not deal with CAS in any way
- "dry"
    - If the package's hash (code unit style) already has a CAS record, return a successful "refactor already applied" status and do not run the refactor.
    - Otherwise, run the refactor, then re-hash / compare files. If code changed, return "successfully applied refactor". If code was unchanged, return "no refactoring opportunities found".
    - Certify the post-run hash as DRYed either way. Subagent message is explanation only.

In general, I want to VERY EASILY add a new prompt-style refactor type:
- control panel is the SPEC.md.
- inputs into a prompt-style refactor:
    - name
    - prompt (Markdown file in data/)
    - agent name (e.g., "limited_package_mode")
    - CAS policy. E.g., none, or code-unit are the two values for now.

User display of the tool (in TUI/noninteractive):
- for prompt-style, normal nested events. Keep final message displayed there, don't hide it.
- for "docs-add":
    - I guess optimally we stream stdout in the same way we do for codalotl_cli. I don't want to duplicate that logic here though. Somehow add a dependency to codalotl_cli, or factor our shared helper?
- overall tool call result
    - successfully applied refactor
    - no refactoring opportunities found
        - for "dry", this means the refactor ran but found nothing useful to DRY up.
    - refactor already applied
        - for CAS-backed refactors, this means the current code unit already has a CAS record.
    - error: xxx

You must use these phases:
phase 0: understand
- run something like `go run . exec --package internal/something "use change_api to request changes to internal/bar to ___."` - keep the task small. Don't commit what this does (if anything). This is just for you to see what it looks like when a tool call spawns a subagent.
- run something like `go run . exec "use codalotl_cli to run add docs on the internal/gousage package"` - internal/gousage is a good candidate because it's small and won't burn many tokens. Don't commit result.
- Add the output of both to this PR file.

Phase 1 to N-1: you figure it out

phase N: validate
- run something like `go run . exec "use the 'refactor' tool to add docs to the internal/gousage package"` (again, don't commit the result)
    - no CAS files should be touched
- run something like `go run . exec "use the 'refactor' tool to DRY up the internal/something package"` (pick a small package that might need some DRYing; don't commit result)
    - should create a CAS file
- See if that looks good and follows our sensibilities and actually works.
- Iterate back to previous phases until this is good.
