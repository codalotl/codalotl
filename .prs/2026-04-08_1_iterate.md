# PR

## User Summary (do not modify)

Overall problem statement: we just shipped /orchestrate, and it's decent - it's a workflow/state machine that needs to be "called" many times to finally get to terminal state (ex: first pass: edit pr file. second pass: iterate something. third pass: edit pr file again. etc). I want a 1-command thing to leave it cooking for an hour to see where it ends up. See internal/agentbuilder/data/pr-orchestrator.prompt.md.

Related problems: I have written other, similar iterative prompts like this (for various purposes - maintain a wiki; do research; etc) - ones that are not hard-coded into codalotl. I want a way to let those cook as well.

Related to that, one key requirement: I want the ability to burn tokens not with codalotl's API keys, but with, for example, codex's subscription plan. The coordinator can still use the codalotl api key, but the bulk of the work is done with codex. (Again, this is just the **option** to do this; this also needs to work with codalotl itself, if a user has no codex, for instance). This may be achieved "for free" with simple shell commands, but I want to call it out as a requirement now.

In terms of "continuing iteration", there's 2 ways:
- fresh session (new context). This is a good clean default, and works well when each step modifies some external state (eg, wiki, filesystem, etc). Good when a lot of context window is consumed after an expensive step.
- resume session (re-use context). Instead of making a fresh session, we can just send a "continue your work" message to the agent on existing thread. This re-uses its context. Good when steps have a big setup but not much actual work. Good when not much context is used yet.

Sketch of solution:

```bash

# general solutions:
codalotl iterate --prompt-file="/path/to/prompt.md"
codalotl iterate "prompt is right here"

# orchestrate-specific:
codalotl iterate --orchestrate "use .prs/2026-04-08_2_iterate.md" # the prompt ("use .prs/2026-04-08_2_iterate.md") is optional - orchestrate in general works without a user message (it can find the PR file itself).

# max steps:
codalotl iterate --max-steps=10 "prompt is right here"

# max minutes (if execution time is greater than this, it won't start a new iteration. but it doesn't hard-kill in-progress step):
# NOTE: this IS compatible with --max-steps
codalotl iterate --max-minutes=30 "prompt is right here"

# custom decision prompt:
codalotl iterate --decision-prompt="output STOP_ITERATION if the workflow is done, otherwise output CONTINUE_ITERATION" "prompt is right here"

# --continue-mode supports fresh, resume, and auto. Default is auto.
codalotl iterate --continue-mode=fresh "some prompt"
```

What does this actually do?
- Uses generic agent (and its prompt). the "prompt" is just the first user message. for orchestrate, i think we have an orchestration agent. (see agentbuilder).
- Each iterate step might use its own session, or re-use the previous iteration session (see continue mode).
- Runs the agent in noninteractive mode.
    - iterate accepts relevent exec CLI flags, like --yes, --json, --model, --no-color, --slash-command, but NOT --package.
    - Output agent stream to stdout, as noninteractive does today.
    - But also output to stdout metadata before and after the step. eg, "iteration 3: starting" and "iteration 3: success, decision=continue"
        - This is high level idea, you can wordsmith. Also handle --json.
- Then we have to decide if we're going to do another iteration.
    - there's max-steps and max-minutes terminal conditions
    - Else If the stream ended with !EventTypeDoneSuccess (eg, EventTypeError):
        - K=3 retries left: continue
        - else: stop.
    - Else If the final message contains STOP_ITERATION (simple substring check), then stop.
    - Else If the final message contains CONTINUE_ITERATION or CONTINUE_FRESH or CONTINUE_RESUME, then continue.
    - Else If there's a decision prompt (there IS by default, even if not specified by user. Only blank if --decision-prompt="")
        - Send decision prompt as a new message to existing conversation.
        - bad stream exit: same as above (share retry counter)
        - We then check for STOP_ITERATION and such again.
        - else: continue iteration.
    - Else, continue.
    - if we decide to continue, we can continue with fresh or resume mode.
        - priority 0 (lower=more priority): --continue-mode of fresh or resume
        - priority 1: CONTINUE_FRESH or CONTINUE_RESUME present
        - priority 2: if context is 25% used or less, resume
        - priority 3: fresh

The default decision prompt looks similar to "output STOP_ITERATION if the workflow is done, otherwise output CONTINUE_ITERATION", but probably is a bit longer with more generic hints, but not TOO much longer. Needs wordsmith.
- I don't think the default decision prompt needs CONTINUE_FRESH vs CONTINUE_RESUME, that's more of an advanced optimization

NOTE: ctrl-c from user must break out of iteration and end program, not merely stopping current step (and possibly starting again).
