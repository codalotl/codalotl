# PR

## User Summary (do not modify)

Overall problem statement: we just shipped `/orchestrate`, and it's decent. It's a workflow/state machine that needs to be "called" many times to finally get to a terminal state (e.g., first pass: edit PR file; second pass: iterate something; third pass: edit PR file again; etc.). I want a one-command way to leave it cooking for an hour to see where it ends up. See `internal/agentbuilder/data/pr-orchestrator.prompt.md`.

Related problems: I have written other similar iterative prompts like this for various purposes (maintain a wiki, do research, etc.) that are not hard-coded into codalotl. I want a way to let those cook as well.

Related to that, one key requirement: I want the ability to burn tokens not with codalotl's API keys, but with, for example, Codex's subscription plan. The coordinator can still use the codalotl API key, but the bulk of the work is done with Codex. Again, this is just the **option** to do this; this also needs to work with codalotl itself if a user has no Codex, for instance. This may be achieved "for free" with simple shell commands, but I want to call it out as a requirement now.

In terms of "continuing iteration", there are 2 ways:
- fresh session (new context). This is a good clean default, and works well when each step modifies some external state (e.g., wiki, filesystem, etc.). Good when a lot of context window is consumed after an expensive step.
- resume session (re-use context). Instead of making a fresh session, we can just send a "continue your work" message to the agent on existing thread. This re-uses its context. Good when steps have a big setup but not much actual work. Good when not much context is used yet.

Sketch of solution:

```bash

# general solutions:
codalotl iterate --prompt-file="/path/to/prompt.md"
codalotl iterate "prompt is right here"

# orchestrate-specific:
codalotl iterate --orchestrate "use .prs/2026-04-08_1_iterate.md" # the prompt ("use .prs/2026-04-08_1_iterate.md") is optional - orchestrate in general works without a user message (it can find the PR file itself).

# max steps:
codalotl iterate --max-steps=10 "prompt is right here"

# max minutes (if execution time is greater than this, it won't start a new iteration, but it doesn't hard-kill an in-progress step):
# NOTE: this IS compatible with --max-steps
codalotl iterate --max-minutes=30 "prompt is right here"

# custom decision prompt:
codalotl iterate --decision-prompt="output STOP_ITERATION if the workflow is done, otherwise output CONTINUE_ITERATION" "prompt is right here"

# --continue-mode supports fresh, resume, and auto. Default is auto.
codalotl iterate --continue-mode=fresh "some prompt"
```

What does this actually do?
- Uses a generic agent (and its prompt). The "prompt" is just the first user message. For orchestrate, we have an orchestration agent (see `agentbuilder`).
- Each iterate step might use its own session, or re-use the previous iteration session (see continue mode).
- Runs the agent in noninteractive mode.
    - iterate accepts relevant exec CLI flags, like `--yes`, `--json`, `--model`, `--no-color`, `--slash-command`, but not `--package`.
    - Output agent stream to stdout, as noninteractive does today.
    - But also output metadata to stdout before and after the step, e.g., "iteration 3: starting" and "iteration 3: success, decision=continue".
        - This is a high-level idea; you can wordsmith it. Also handle `--json`.
- Then we have to decide if we're going to do another iteration.
    - First, check max-steps and max-minutes terminal conditions.
    - Else if the stream ended with `!EventTypeDoneSuccess` (e.g., `EventTypeError`):
        - If retries remain (max 3 total), continue.
        - Else, stop.
    - Else if the final message contains `STOP_ITERATION` (simple substring check), then stop.
    - Else if the final message contains `CONTINUE_ITERATION`, `CONTINUE_FRESH`, or `CONTINUE_RESUME`, then continue.
    - Else if there's a decision prompt (there is by default, even if not specified by the user; it is only blank if `--decision-prompt=""`):
        - Send the decision prompt as a new message to the existing conversation.
        - On a bad stream exit: same as above (share the retry counter).
        - Then check for `STOP_ITERATION` and related tokens again.
        - Else, continue iteration.
    - Else, continue.
    - If we decide to continue, we can continue with fresh or resume mode.
        - priority 0 (lower = higher priority): `--continue-mode` of `fresh` or `resume`
        - priority 1: `CONTINUE_FRESH` or `CONTINUE_RESUME` present
        - priority 2: if context usage is 25% or less, resume
        - priority 3: fresh

The default decision prompt looks similar to "output `STOP_ITERATION` if the workflow is done, otherwise output `CONTINUE_ITERATION`", but probably should be a bit longer with more generic hints, though not too much longer. Needs wordsmithing.
- I don't think the default decision prompt needs `CONTINUE_FRESH` vs. `CONTINUE_RESUME`; that's more of an advanced optimization.

NOTE: Ctrl-C from the user must break out of iteration and end the program, not merely stop the current step and possibly start again.

## Plan

### internal/iterate
- Add a new iteration coordinator package that runs one agent step at a time until a terminal condition is reached.
- Support prompt text vs `--prompt-file`, `--max-steps`, `--max-minutes`, `--decision-prompt`, and `--continue-mode` (`fresh`, `resume`, `auto`).
- Centralize stop/continue decisions: max-step and max-minute limits, stream-success vs stream-failure retry handling, decision-token parsing, default decision prompting, and fresh-vs-resume selection.
- Keep the step runner abstract enough to support codalotl-native execution by default and an alternate external runner path for users who want iteration work to burn non-codalotl tokens.

### internal/noninteractive
- Add or expose the session/result plumbing needed for iterative execution, especially when `resume` mode reuses an existing conversation.
- Preserve existing one-shot `exec` behavior while making iteration able to inspect terminal outcome, final assistant text, and context-usage data needed for decision-making.
- Cover new behavior with focused tests around success/error/cancel outcomes, resume semantics, and any structured metadata consumed by the iterator.

### internal/cli
- Add `codalotl iterate` with the new iteration-specific flags plus the relevant noninteractive execution flags (`--yes`, `--json`, `--model`, `--no-color`, `--slash-command`), but not `--package`.
- Add the orchestrate convenience mode/flag and wire prompt sourcing so `codalotl iterate --orchestrate` works with or without an explicit user message.
- Print iteration lifecycle metadata before/after each step in text and JSON modes, and ensure Ctrl-C stops the loop rather than merely ending the current step.
- Update CLI tests and `SPEC.md` for the new command surface.

### Docs and validation
- Update any user-facing docs that describe noninteractive execution so `iterate` is discoverable and its stop/continue behavior is clear.
- Manually test representative flows: fresh vs resume, auto mode, max-step and max-minute termination, explicit continue/stop tokens, retry-after-failure behavior, and orchestrate-driven iteration.

## Decisions

- Prefer a dedicated coordinator package for iteration logic instead of embedding the loop directly in `internal/cli`, so the policy and stop conditions stay testable.
- Treat external-token execution as a first-class requirement in the design: even if the first implementation path uses codalotl-native execution, the iterator should not hard-code itself to only one backend/session source.
